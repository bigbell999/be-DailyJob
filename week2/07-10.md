# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - 서비스 계층 도입 확정. 계획, 근거 수립.
  - 정적 리소스 핸들러의 이원화로 구현한 실제 index.html 경로 접근 차단.
  - html renderer의 구현
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - model 클래스와 repository 클래스의 책임 분리 구현
  - Service 계층 구현
  - 구체적 예외 구현

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 비즈니스 로직을 다루는 Service 계층의 분리
- 그렇게 판단한 이유:
  - 지금까지 (6단계)는 비즈니스 로직이 복잡하지 않아서 handler 내부에서 비즈니스 로직을 처리해도 괜찮았다.
  - 그러나 7단계부터는 User, Article, Comment 등 다뤄야 할 도메인이 늘어나면서 handler 가 너무 두꺼워진다고 판단했다.
- 고려했지만 선택하지 않은 대안:
  - User 도메인에 대한 것은 현상을 유지한다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 만약 User 도메인을 다루는 것을 현재처럼 모델이 한다고 생각했을 때, 추후 Article과 Comment 가 Service 계층으로 구현되어 handler 안에 혼재되었을 때, 코드적으로 지저분하고 혼란을 줄 수 있다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - 정적 리소스 디렉토리에 있는 assest들이 로딩이 안됨.
- 원인
  - 논리 주소만을 사용하기 위해서 fallback handler가 실제 경로를 받는 것을 거부하도록 코드를 작성함.
- 해결 과정
  - 처음에는 index.html 만을 다루는 핸들러를 추가할까 생각함.
  - 그러나 index.html 들은 비즈니스 로직과 밀접하게 연결되어 있음.
  - 따라서 fallback handler를 파라미터를 기준으로 이원화 시킴 (asset only - index.html 서빙 요청은 거부, 이를 fallback 으로 등록하여 정적 리소스 반환)


### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - Service 계층을 도입할 것인지에 대해 claude와 논의 중, claude가 제시한 근거를 읽고 다시 정리함
- 그 접근이 도움이 된 이유:
  - 정리하는 과정에서 claude가 사용하는 용어에 대해 공부하게 되었으며, 근거를 이해하고 반박하는데 도움이 됨.
  - AI의 의견을 아 그런갑다하고 넘어간 적이 솔직히 많았었는데, AI의 의견을 이해하는 과정에서 새로운 지식을 이해할 수 있는 능력을 기룰 수 있게 됨.
- 다음에도 반복하고 싶은 점:
  - 머리로만 이해하는게 아니라 AI의 의견을 내가 정리하고 다시 검토를 맡김으로써 내 이해를 단단하게 한다.
