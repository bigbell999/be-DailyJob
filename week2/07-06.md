# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - dispatcher - router - handler - parser 계층 구조 구현
  - HttpRequest, HttpResponse 레코드 구조로 handler의 입출력 통일.
  - 바이트 스트림을 읽어 \r\n 단위로 끊어 저장하는 파서 구현
  - shared mutable state인 Database에 회원을 synchronized로 저장하는 메서드 구현. + 회원 ID, 닉네임 중복 검사
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - claude code 의 /skills와 /hook을 사용해보지 못한 것.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 계층 구조를 확정하고 handler의 입출력을 HttpRequest와 HttpResponse로 확정한 것.

- 그렇게 판단한 이유:
  - http 요청은 request line, header, body 로 구조화된 객체이다.
  - handler는 요청을 처리하고 http response를 반환하는 객체이다.
  - 따라서 handler는 HttpRequest라는 객체만 있으면 Http 상의 모든 동작이 처리 가능하다.

- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - method, path를 handler 와 매칭하여 라우팅하기로 이전에 결정하였기 때문에 handler interface를 구현하므로써, 이후의 handler 개발이 편해졌다.


### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - HttpRequest 객체를 만드는 parser 를 구현하기가 어려웠다.
  - request line과 header를 BufferedInputStream으로 readLine으로 읽어버리면 추후 body에 image등 text가 아닌 값이 오는 경우 내용이 손상될 수 있다.
  - 따라서 InputStream을 바이트 단위로 읽어야 했다.
  - 이때 \r 을 만난 경우 뒤에 \n이 있거나 바이트가 없는 경우 어떻게 처리해야 할 지가 문제였다.

- 문제 해결 과정
  - 먼저 \r이 나왔는데, \n이 나오지 않는 경우는 예외처리해야 하는 상황임을 인지했다.
  - 그리고 예외가 발생할 수 있는 케이스를 정리했다.
  - 1. \rEOF : 인터넷 연결 문제 일 수 있음 - 추후에 custom Exception을 추가할 예정이다. //TODO를 통해 마킹했다.
  - 2. \r (\n이외의 다른 바이트) : 요청 형식 문제 - 400 Bad Request

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - 구조를 나누면서 그 동작에 필요한 데이터 구조를 먼저 생각하였다.

- 그 접근이 도움이 된 이유:
  - 필요한 데이터 구조를 먼저 생각하다 보니 기존에는 이름만 같던 파서와 핸들러의 입출력이 통일 되었다.

- 다음에도 반복하고 싶은 점:
  - 시스템의 구조를 설계할 때에 입출력에 필요한 데이터를 먼저 정의한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 회원가입 핸들러 테스트 코드를 작성하는 과정에서 내가 생각한 테스트 케이스를 claude code에게 던지고 테스트 코드 작성시켰다.
  - 이 때 오늘 배운 /hook을 적극적으로 사용할 수 있었는데, 그러지 못했다.

- 그렇게 된 원인:
  - InputStream과 BufferedInputStream 실험한다고 시간이 없었다.

- 다음에는 다르게 해볼 구체적인 행동:
  - 지금 만드는 프로젝트를 claude code로만 작성하면서 claude code의 /hook과 /skills 에 대해 학습한다.
