# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - get 으로 회원가입 기능 작성
  - shallow/deep immutablity
  - 로버스트니스 원칙

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 쿼리 문자열을 파싱할 때 split 을 사용한다.
- 그렇게 판단한 이유:
  - 쿼리 문자열은 가변적이기 때문이다. 그리고 가변적인 상황에서 틀린 상황이 없기 때문이다. ? 없이 오는 path도 틀린 요청이 아니기 때문.
- 고려했지만 선택하지 않은 대안:
  - 정규식을 사용한다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 정규식을 사용했다면 정규식 컴파일 비용과 조건 분기가 더 복잡해졌을 것이다.

### 2-1. 오늘 가장 의미있었던 기술적 판단
- 어떤 행동이었는지:
  - 파싱된 request line 을 받아 처리하는 클래스의 이름을 모두 handler 라고 통일하고 methodHandler 를 상속 받게 만들었다.
  - 이로써, 핸들러가 return 하는 자료형이 필요하여 HttpResponse 레코드를 만들었다.
- 그 접근이 도움이 된 이유:
  - 핸들러를 호출하는 부분은 모두 공통된 처리가 가능하고 핸들러와 호출부의 결합도가 낮아졌다.
