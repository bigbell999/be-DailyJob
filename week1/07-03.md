# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - 기능
    - 서버가 여러가지 mimeType의 자료들을 다룰 수 있는 기능 추가(gozila 문서 참조)
  - 학습 
    - thread-per-connection의 개념과 문제점에 대해서 학습
  - 결정
    - 서버 런타임 중 접근하는 리소스의 상한선(base) 정의.
    - 서버 구조 정의와 계층 정하기
      - worker - processor - handler/parser
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 파서가 안정적으로 http 첫 라인을 파싱하는 방법.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 어제 내린 판단에 대해서 구현만하여 기술적 판단을 할 시간이 없었습니다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - contentType을 결정하는 것은 processor에서 해야하나 handler에서 해야하나?
- 최초 접근
  - 각 클래스의 책임에 대해서 글로 작성해본다. 아무 컨텍스트도 없는 LLM에게 설명할 정도로 구체적으로 적는다.
  - Processor
    - http 요청을 받아 적절한 handler와 parser를 호출하고 결과를 조립한다. 그리고 예외도 처리한다.
  - handler
    - 특수한 작업(method 해석, socket에 데이터 쓰기) 등을 작업한다.
- 이렇게 역할을 분리해서 글로 적어보니, 당연히 다루는 데이터의 타입 분류는 handler에서 해야한다는 결론에 도달할 수 있었다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - 이 '행위'를 어떤 클래스가 해야할 지 결정할 때, 그 클래스의 책임과 역할을 글로 작성해본다.
- 그 접근이 도움이 된 이유:
  - 글로 작성하면서 클래스의 구조, 설계에 문제가 생겼음을 확인할 수도 있고 텍스트이기 때문에 LLM에게 검사를 받아볼 수도 있다.
  - 결정적으로는 글로 작성한다는 행위가 생각이 정리된다.
- 다음에도 반복하고 싶은 점:
  - 이렇게 애매할 때마다 작성한 글이 모여 클래스/모델 설계에 도움이 될 것 같다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 조를 섞어 진행했던 세션에서 분명 이상한 점이 있었으나 그 점을 명확히 알지 못한채 떠들어 대었다.
- 그렇게 된 원인:
  - 나중에 알게 된 사실이지만, thread-per-connection 문제라는 것을 알았다.
  - 사람들과의 토론을 멈추고 차분히 문제를 먼저 정의했다면 검색이나 질문으로 빠르게 해결할 수 있었을 것이다.
- 다음에는 다르게 해볼 구체적인 행동:
  - 문제가 있는 것은 알겠는데, 문제가 무엇인지 정의조차 못하겠다면 일단 말을 하지 않는다.
  - 차분히 코드를 읽으며 생각하고 의심이 되는 부분을 마킹했다가 강사님께, 혹은 LLM에게 질문하겠다.
