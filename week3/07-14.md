# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - 멀티 파트 요청을 받기 위해 http요청의 body변경: requestLine/headers/body -> requestLine/headers/body(원문)/fields/files
  - 멀티 파트로 오는 http 요청 파싱로직 확장.
  - 이미지 서비스, 이미지 레포지토리 구현
  - 홈 화면 핸들러가 이미지 서비스/ 보드 서비스를 사용하여 유저 프사, 게시글 사진을 로드하도록 변경

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 멀티파트 content type 처리를 위해서 HttpRequest의 객체 구조를 변경하였다.
- 그렇게 판단한 이유:
  - 만약 객체 구조를 변경하지 않았다면, httpRequest를 파싱하는 파서와 이미지 비즈니스 로직의 결합도가 올라가 추후 8번 과제를 진행하는데에 어려움이 있었을 것이다.
- 고려했지만 선택하지 않은 대안:
  - httpRequest의 구조는 유지하되, 입력 스트림을 가지는, 그러니까 stateful한 파서를 하나 추가하여 처리한다.
  - stateful한 파서가 이미 존재한다. (raw byte -> http request) 이 파서에 content type을 분기하여 처리한다고 해도 http request에 파일을 담을 방법이 없다.
  - 따라서 http request의 구조를 변경하여 파일을 담을 수 있도록 한다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - http request가 god-context 가 되었기 때문에 모든 핸들러가 공통적으로 사용할 수 있다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - 이미지를 어떤 레이어에서 다루어야 하는가?
- 처음 추측:
  - 이미지 업로드 요청 (POST) 를 처리할 수 있는 핸들러를 하나 추가한다.
- 원인을 확인하기 위해 시도한 방법:
  - claude code 의 plan mode를 통해서 핸들러를 추가할 경우에 새로 작성하거나 추가해야할 점에 대해 질문하였다.
- 해결 방법:
  - 만약 이미지 POST 처리를 위한 핸들러를 추가한다면 그 핸들러의 처리 결과로 발생한 image 경로를 HomeHandler가 처리해야 한다.
  - 이를 위해서 핸들러가 핸들러를 호출하는 일이 발생하는데 이를 오류라고 판단하였다.
  - 따라서 이미지를 업로드는 비즈니스 로직이라고 판단하였고 서비스 레이어에 추가하였다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - claude code 세션의 컨텍스트가 길어지면서 성능 저하가 발생하였음.
  - plan으로 저장한 후 다른 세션에 주입함
- 그 접근이 도움이 된 이유:
  - 컨텍스트를 유지하면서 토큰 사용량과 성능을 챙길 수 있었다.
- 다음에도 반복하고 싶은 점:
  - 작업을 지시하거나 논의를 할 때, plan 모드를 활성화 시켜 작업한다.
