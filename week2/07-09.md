# be-DailyJob

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
    - 인증 처리를 데코레이터로 일원화했음
      - LoggedinOnlyHandler 도입으로 통합, 세션 인증이 필요한 핸들러를 모두 래핑해 라우터 등록.
    - 검증한 세션을 타입으로 전달
      - 이 전엔 핸들러에는 한가지 타입만 있었는데, 인증이 필요한 핸들러 내부에서 검증 로직이 반복됨.
    - 세션 로직 객체화
      - SessionUtil → 주입 가능한 Authenticator로 승격(발급/조회/폐기 책.
    - 전역 static DB를 Repository 인터페이스(UserRepository/SessionRepository)로 추상화하고 DB가 implements하도록 변경.
    - 반환값 규약 정립: 조회 계층을 null → Optional로 통일.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
    - 정적 파일 보안(인증 대상 템플릿 HTML의 fallback 노출)

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 전역 static DB를 직접 호출하는 대신, Repository 인터페이점에서 주입하는 구조로 전환.
- 그렇게 판단한 이유: 7단계에서 실제 DB를 적용하기 때문에. 이때 실제로 바뀌는 것은 구현이므로, 그 경계에 인터페이스가 있어야
  나머지 코드가 흔들리지 않는다는걸 몸으로 배움.
- 고려했지만 선택하지 않은 대안
    - static 유지: 주입, 교체, 테스트가 막혀 DB 전환 시 전역이 오염됨.
    - god context 객체를 넘겨 핸들러가 꺼내 쓰기
- 품질/테스트/유지보수에 미친 영향: 
  - 의존 관계가 생성자 시그니처에 드러나고, DB 전환이 구현 클래스 한 줄 교체 + 조립부 수정으로 간단해짐.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황/증상: 로그아웃을 구현하려는데, 세션을 폐기할 sid를 얻을 방법이 없었다.
- 처음 추측한 원인: 핸들러 시그니처의 문제라고 봤다.
- 원인 확인 시도: 데코레이터가 세션을 어떻게 resolve해 델리게이트로 넘기는지, Session 객체가 무엇을 보유하는지 추적.
- 실제 원인: Session이 자신의 sid를 갖고 있지 않았다. 왜냐하면 이전 단계에서 session db가 키값으로 sid를 사용했기 때문이다.
- 해결 방법: Session에 sid를 부여하고, 발급/폐기를 Authenticator.login/log는 세션에서 sid를 얻어 폐기.
- 다음에 먼저 확인할 것: "도메인 객체가 자기 식별자를 스스로 갖고 있는가?" 검증한 값을 사용하는 쪽까지 통로가 이어져 있는가(검증-사용 분리 여부).
