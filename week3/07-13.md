### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 댓글 목록의 반복 렌더를 조각 템플릿 + 렌더러가 합치는 방식으로 구현했다. `comment/item.html` 조각을 댓글마다 렌더해 이어붙이는 `renderEach`를 `TemplateRenderer`에 추가하고, 핸들러는 도메인 데이터를 뷰 모델로 변환만 한다.
- 그렇게 판단한 이유:
  - 검증은 Service, HTML 조립은 뷰, HTTP 변환은 핸들러라는 계층 경계를 지키기 위해서였다. 조각 렌더를 뷰 계층(`renderEach`)에 두면 HTML 조립이 핸들러로 새어나가지 않는다.
  - 값 이스케이프(XSS)도 렌더러가 조각 필드에 자동 적용할 수 있어 보안 처리를 한 곳에 모을 수 있었다.
- 고려했지만 선택하지 않은 대안:
  - 핸들러가 댓글 HTML 문자열을 만들어 단일 placeholder에 주입** — 가장 빠르지만 HTML 조립이 핸들러로 새어 계층 경계가 무너지고, 이스케이프를 수동 관리해야 함.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 이스케이프가 렌더러/조각에 모여 있어, 사용자 입력으로 렌더되는 것을 검증으로 확인할 수 있었다.
### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - 회원가입, 로그인, 글쓰기는 302로 정상인데, 댓글, 좋아요만 전부 400이 떨어졌다.
- 처음 추측한 원인:
  - DB에서 추출한 articleId 값이 이상하거나, 세션 인증이 안 된 것으로 추측했다.
- 원인을 확인하기 위해 시도한 방법:
  - 상태 코드가 401이 아니라 400 이라는 점에서, 로그인 게이트는 통과했고 핸들러 내부에서 `IllegalArgumentException`이 났다고 범위를 좁혔다.
- 실제 원인:
  - `parseParams`가 파라미터 키를 `toLowerCase()` 로 저장한다. 그래서 body의 `articleId`가 맵에는 `articleid`로 들어갔다.
- 해결 방법:
  - body를 받을 때 전체를 lowercase한 키로 코드를 수정했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 400이 나면 파싱/`IllegalArgumentException` 계열부터 의심하고, 값이 null로 들어오는지(맵 키 불일치) 확인.
