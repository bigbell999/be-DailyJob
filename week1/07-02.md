# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - java의 record에 대해서 학습
  - java의 thread가 어떻게 동작하는지. 
  - thread.start()와 thread.run() 이 어떻게 다른지.
  - 클래스를 설계할 때 필드 보다는 행위(메서드)를 먼저 정의해야 한다는 점. 이는 책임 분리를 위함.
  - OOP의 SRP의 개념.
  - try-with-resource의 사용 방법과 close되는 시점.
  - DDD와 어플리케이션 중심 개발론의 특성 비교.
  - 식별/비식별 관계
  - DA 강의 쇼핑몰 상황 ERD 예제

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - record를 사용하여 request line의 method, path, version을 저장하는 데이터 구조를 정의한 것.
- 그렇게 판단한 이유:
  - 데이터 저장을 위해서 별도의 데이터 클래스를 작성할 필요가 없고, priavte final로 데이터가 저장되어 변경될 우려도 없음.
- 고려했지만 선택하지 않은 대안:
  - 데이터 클래스를 새로 정의하려 했지만 http header전체를 저장하는 데이터 클래스가 현 단계의 과제에서는 필요없다고 판단함.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 별도의 데이터 클래스를 작성하지 않아서 시간이 단축되었다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - ERD 예제에서 '회원 엔티티'가 여러개의 배송지를 가질 수 있고, 배송지 중에서 '기본 배송지'를 설정할 수 있는 상황이었다.
  - 해당 상황에서 2가지 방안이 있었다.
  - 1. 배송지 엔티티를 회원에 종속시킨다. (1: N 관계) 그 후에 배송지 엔티티 속성에 '기본 배송지'를 boolean값으로 추가한다.
  - 2. 전국의 모든 주소를 저장하는 주소 엔티티를 만들고 회원 엔티티와 N:M 관계로 설계한다. 그리고 배송지라는 엔티티로 둘을 연결한다.

- 2번이 헷갈렸던 이유.
  - 주소지는 고유한데, 1:N 관계로 회원과 배송지를 엮어 버리면 DB크기가 커져 낭비될 것이라고 생각함.
  
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 다른 속성을 고려하지 말고, pk와 fk만 생각하고 엔티티간의 관계에 집중할 것.
  - 칸 방식의 다이어그램으로 먼저 그려볼 것.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - SRP를 공부하고 공부한 내용을 적용하여 http request의 첫라인을 파싱하는 유틸 클래스의 책임을 분리한 것.
  - 데이터 저장은 record 에게 책임을 주고 파서는 파싱에만 책임을 지도록 코드를 수정함.
- 그 접근이 도움이 된 이유:
  - 이전 버전의 파서는 파싱한 결과를 내부 필드에 저장하여 getter 메서드로 필드의 method, path, version에 접근하게 하였음.
- 다음에도 반복하고 싶은 점:
  - 어떤 유틸 클래스를 작성할 때, 행위(책임)을 먼저 생각하는 습관.
