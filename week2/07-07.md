# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - ConcurrentHashMap 과 synchronized 가 각각 데이터 정합성을 지키는 방법
  - Stateless한 클래스들은 한 클래스에 모아서 static 한 메서드로 만드는 개발
  - Secure Random의 사용법.
  - Session ID Collision이 유의미해지는 바이트의 길이
  - Session 만료 정책과 Active/lazy 무효화 방법과 그 구현.
  - 사용자의 마지막 행동을 Session 내의 저장된 필드값을 이용해 저장하는 법.

- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - URI 라이브러리를 사용하여 Http Request Line 의 target을 분리하는 법.
  - nullable 객체의 의미와 처리 방법.
  - 사용자의 세션에 따라 동적으로 변하는 html

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 세션 무효화를 판단할 때 언제, 누가 확인할 것인가?
  - 세션 ID를 통해서 DB에 저장된 세션 정보에 접근할 때 만료를 판단한다. (IDLE)
- 그렇게 판단한 이유:
  - 현재 ConcurrentHashMap으로 Session DB가 구현되어 있기 때문에 언젠가는 실제 DB를 사용한다.
  - 이때 Active한 세션 DB 감시 클래스가 있다면 이것까지 리팩토링 해야한다.
- 고려했지만 선택하지 않은 대안:
  - 백그라운드에서 세션 DB를 계속 감시하는 작업을 하나 만든다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 실제 DB가 연결되었을 때 리팩토링할 비용이 줄어든다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - stateless 한 parser 클래스가 여러개 있어서 정리가 필요했다.
  - /skills와 /plan, /hook 을 사용하여 claude code로 리팩토링을 진행함.
- 그 접근이 도움이 된 이유:
  - /plan 기능을 통해 리팩토링이 어떻게 진행될 지 예상할 수 있었고 그 이후의 테스트 까지의 작업들을
  - /hook과 /skills를 통해서 자동화 할 수 있었다.
  - 리팩토링은 프로젝트 규모가 커질 수록 비용이 기하급수적으로 늘어난다.
  - 이러한 작업은 특별히 기술적으로 고민이 끝난 문제를 묶는 단순한 작업이지만
  - 사람이 하게 되면 실수할 확률이 높아진다.
- 다음에도 반복하고 싶은 점:
  - 병합 수준의 단순한 리팩토링은 AI에게 맡기겠다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 지금까지 http request line의 target을 URLDecoder로 디코딩 중이었다.
  - path 디코딩과 파싱을 하는 라이브러리가 이미 존재했다.
- 그렇게 된 원인:
  - 과제를 쳐내기에만 급급하여 자료 조사가 부족했다.
- 다음에는 다르게 해볼 구체적인 행동:
  - 직접 기능을 구현하기 전 이미 개발된 레퍼런스나 라이브러리, 공식 문서를 찾는 데에 시간을 더 쓰는 것이
  - 오히려 시간이 덜 들 수도 있다.
